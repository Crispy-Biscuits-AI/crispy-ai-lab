OpenBrain Multi-Chunk Metadata Test File

This file exists only to test the OpenBrain ingest workflow in n8n.

The purpose of this test is to force the workflow to create multiple chunks so that we can verify:
- chunk_index increments correctly
- total_chunks is identical across all rows
- the filename and ingested_at metadata remain consistent

Section 1

OpenBrain is a self-hosted AI laboratory built incrementally. The system currently includes n8n running in Docker Desktop, Ollama running locally, and Postgres with pgvector. The workflow already reads a text file, splits it into chunks, and inserts those chunks into the memories table. The next goal is to prove that metadata remains correct when the workflow creates more than one chunk.

Section 2

The metadata test should produce several inserted rows. Each row should preserve the original source, category, title prefix, and content. The only difference between rows should be the chunk title suffix and the metadata_json values. The first row should contain chunk_index equal to one. The second row should contain chunk_index equal to two. The third row should contain chunk_index equal to three.

Section 3

The value total_chunks should be the same on every row created from this file. If the workflow creates three chunks, then each row should report total_chunks equal to three. If the workflow creates four chunks, then every row should report total_chunks equal to four. This test is important because future embedding generation and retrieval behavior will depend on correct chunk tracking.

Section 4

The filename stored in metadata_json should be exactly the filename of this test file. The ingested_at field should be an ISO timestamp created during workflow execution. All chunks produced in a single run should share the same ingested_at value.

Section 5

This paragraph intentionally adds more text so that the file becomes larger than the current chunk_size setting of twelve hundred characters. The workflow currently uses a chunk size of twelve hundred and a chunk overlap of two hundred. That means the file must be substantially longer than twelve hundred characters to prove that multiple chunks are generated correctly.

Section 6

The OpenBrain ingest workflow should now be able to create several rows in the memories table. After the workflow runs, the SQL validation query should show multiple rows with titles ending in chunk zero one, chunk zero two, and chunk zero three. The metadata should confirm that all rows came from the same original file.

Section 7

This is the final section of the test file. It exists only to ensure the text length is large enough to guarantee multiple chunks. If the workflow still produces only one chunk, then the file is not long enough and additional text should be added until at least three chunks are created.

Additional filler text begins here.

OpenBrain is being built incrementally because small successful steps are easier to validate than large redesigns. Every new feature should be proven in isolation before another feature is added. The current focus is chunk metadata validation. After chunk metadata is proven for multi-chunk files, the next likely milestone will be adding embeddings with Ollama and storing those vectors in pgvector. Until then, the workflow should remain simple, observable, and easy to debug.
